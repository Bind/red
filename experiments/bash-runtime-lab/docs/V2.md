# V2

Known trade-offs in the current MVP and the likely follow-on work if the concept
proves valuable.

## MVP Summary

The current MVP uses a hybrid mechanism:

- `just-bash` for parsing and AST serialization
- runtime-owned AST instrumentation for implicit command boundaries
- real host `bash` for actual script execution
- hook callbacks for journaling cwd, env, exit status, and workspace diffs

This is the right shape for proving the concept quickly, but it leaves several
important trade-offs unresolved.

## Why The MVP Is Acceptable

The MVP is optimized for validation, not purity.

It gives us:

- real CI/CD tools and host binaries immediately
- command-level implicit boundaries
- workspace-layer journaling over real executions
- a low-friction way to observe actual workflows

That is enough to answer the first question: does command-level, layer-like
durability produce useful behavior for pipeline scripts?

## V2 Trade-Offs

### 1. Execution Ownership

In the MVP, we do not fully own execution.

- we own parsing
- we own instrumentation
- host `bash` owns actual shell semantics

Implication:

- command boundaries are injected by source transformation
- we are observing execution through hooks, not driving execution directly

V2 direction:

- move toward interpreter-owned command dispatch
- make hooks native runtime events instead of injected shell functions

### 2. Fidelity Of Boundaries

The MVP wraps simple commands well enough for initial validation, but boundary
semantics are still shaped by the transformed shell script.

Implication:

- some edge cases in pipelines, functions, subshells, and compound commands will
  be limited by the wrapper strategy
- grouped semantics are not first-class yet

V2 direction:

- define explicit execution events for simple commands, pipelines, and grouped
  command regions

### 3. Workspace Snapshot Cost

The MVP uses naive full pre/post workspace snapshots.

Implication:

- expensive for large repos
- repeated hashing and tree walks will dominate runtime cost

V2 direction:

- incremental tracking
- overlay or copy-on-write workspace layers
- content-addressed layer storage

### 4. Replay Is Not Implemented

The initial MVP only journaled. The current replay prototype now restores exact
matching workspace layers for a narrow safe subset, but it is still incomplete.

Implication:

- durability is partially operational
- workspace effects and exit codes can be reused
- full stdout and stderr replay is not yet first-class
- replay is intentionally prefix-only; more granular reuse would need a stronger
  dependency and log model

V2 direction:

- define replay key semantics
- materialize or apply workspace layers
- return recorded stdout, stderr, and exit status from cache with stronger
  fidelity

### 5. External State Is Out Of Scope

The MVP assumes the tracked workspace is the state that matters.

Implication:

- commands with external side effects are only partially modeled
- `docker`, remote APIs, registries, and deploy steps are not truly captured by
  filesystem diffing alone

V2 direction:

- add command policy classes
- distinguish journal-only commands from replayable commands
- model explicit non-workspace dependencies where needed

### 6. Stable Identity Is Still Basic

The MVP uses AST-path-like ids plus visit counts.

Implication:

- good enough for the experiment
- not yet robust against structural script edits or more advanced reuse cases

V2 direction:

- improve stable node identity
- incorporate source spans, normalized command form, and richer dynamic context

### 7. Process Model Is Still Indirect

The MVP captures env and cwd through hook calls around commands.

Implication:

- process state is observed from the shell boundary
- not all shell-internal state transitions are first-class runtime data

V2 direction:

- promote process state transitions into interpreter-owned execution state

## Decision

Do not solve these trade-offs before the concept is proven.

The MVP should stay:

- simple
- real
- slow if necessary
- honest about limitations

V2 work should happen only after we have enough real CI/CD traces to justify the
extra complexity.
